  RAG를 랭체인으로 바꿔야 할까?

  결론: MVP에서는 지금 방식 유지, 배포 직전에 업그레이드 고려

  현재 방식 (TF-IDF 코사인 유사도)

  장점: 빠름, API 비용 없음, Vercel 서버리스에서 파일만 읽으면 됨
  단점: 한국어에 취약 - "열이 나면" 쿼리가 "발열", "고열" 텍스트를 못 찾음. 의학
   문서처럼 동의어가 많으면 검색 품질이 낮음.

  LangChain + 임베딩 방식

  장점: 의미 기반 검색 - "아기가 열이 나면?" → "발열 시 대처" 자동 매칭. 한국어
  의학 문서에서 품질이 확연히 좋음.
  단점:
  - rag_chunks.json을 임베딩으로 재전처리 필요 (OpenAI text-embedding-3-small or
   로컬 모델)
  - 벡터 DB 필요 (Supabase pgvector, Pinecone 등 - Vercel에서 파일로는 불가)
  - 런닝 비용 발생 (검색마다 임베딩 API 호출)

  지금 당장은 TF-IDF로 진행하는 이유

  1. 벡터 DB 셋업 없이 오늘 배포 가능
  2. 챗봇이 GPT-4o-mini로 답변하기 때문에, RAG 검색이 완벽하지 않아도 GPT가 어느
   정도 보완
  3. 먼저 앱이 제대로 돌아가는지 확인하고 품질 업그레이드가 맞는 순서

  나중에 업그레이드할 때 Supabase pgvector + OpenAI embeddings 조합이 Vercel
  배포 환경에서 가장 현실적입니다.
